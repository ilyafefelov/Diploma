# PRD: Operator MVP Slices For Tenant-Aware Baseline Preview

> *This was generated by AI from an approved planning discussion.*

## Problem Statement

Поточний MVP уже має working control-plane base для Tenant Registry, weather run-config preview, weather materialization і operator dashboard shell. Але далі розвиток ризикує розповзтися в кілька боків одночасно: dashboard framework work, full digital twin ambitions, market-bid semantics і broad backend expansion. Без зафіксованого PRD команда легко змішає upstream control surfaces, baseline LP preview, майбутній `Proposed Bid` domain і dispatch-time safety logic в один нечіткий backlog.

Потрібен короткий, жорсткий product/engineering contract, який фіксує:

- delivery order для перших vertical slices
- межі між operator preview і market-domain semantics
- канонічні read models та status semantics
- мінімальну battery-state simulation модель для LP preview
- guardrails проти scope creep

## Solution

Будуємо MVP через два канонічні vertical slices в жорсткій послідовності.

### Slice 1

Tenant Registry -> weather run-config -> weather materialization -> operator dashboard preview.

Definition of done:

- working end-to-end flow
- operator-facing read models
- documented API contracts
- demo-ready narrative для weekly report / supervisor demo

### Slice 2

DAM baseline forecast -> LP Baseline Strategy -> feasible baseline recommendation preview in dashboard.

Definition of done:

- baseline forecast visible in dashboard
- feasible hourly signed MW schedule visible in dashboard
- projected SOC trace visible in dashboard
- UAH economics visible in dashboard, including degradation penalty as UAH component

### Explicit boundaries

- Slice 1 status semantics live in backend read models, not in frontend composables
- Slice 1 stores latest operator-visible state in Postgres by `tenant_id + flow_type`
- Slice 2 stops at feasible economic recommendation preview and does not yet emit `Proposed Bid`
- Slice 2 uses a `Projected Battery State` simulator, not a full digital twin

## User Stories

1. As an operator, I want to browse the Tenant Registry, so that I can choose a site for an MVP run.
2. As an operator, I want to see the resolved location for a selected tenant, so that I know which weather context will be used.
3. As an operator, I want to prepare a weather run config before execution, so that I can confirm the control-plane payload.
4. As an operator, I want to materialize the weather slice from the dashboard, so that I can trigger the upstream MVP path without using internal Dagster tools directly.
5. As an operator, I want the dashboard to show stable status semantics like `idle`, `prepared`, `running`, `completed`, and `failed`, so that the UI does not invent its own state machine.
6. As an operator, I want the latest visible result for each tenant and flow to survive API restarts, so that demos and follow-up review are stable.
7. As an operator, I want the dashboard to read status from backend read models, so that UI logic stays presentational.
8. As an operator, I want a signal preview surface separate from control actions, so that I can inspect analytical context without mixing it with orchestration buttons.
9. As a developer, I want a lightweight Postgres-backed operator status store, so that read models can be queried consistently across flows.
10. As a developer, I want `flow_type` to be an explicit contract, so that the operator state store can grow slice-by-slice without becoming a blob.
11. As an operator, I want Slice 2 to show a baseline forecast for the selected tenant, so that I can inspect the baseline input before any market semantics are introduced.
12. As an operator, I want Slice 2 to show a feasible signed MW schedule by hourly DAM slot, so that I can understand the LP recommendation in physical terms.
13. As an operator, I want Slice 2 to show projected SOC over the same horizon, so that I can see whether the recommendation stays within limits.
14. As an operator, I want Slice 2 economics to be shown in UAH, so that the preview matches the project's canonical currency.
15. As an operator, I want degradation to appear as a UAH penalty component, so that economics are consistent with the baseline objective.
16. As a developer, I want a `Projected Battery State` simulator driven by hourly DAM slots, so that baseline preview remains aligned with `Level 1 Market Scope`.
17. As a developer, I want simulator inputs to come from the latest known persisted asset state with an optional scenario override, so that offline demos do not block progress.
18. As a developer, I want the simulator to track SOC limits, throughput, and degradation, so that preview feasibility and economics are meaningful.
19. As a domain modeler, I want Slice 2 to stop before `Proposed Bid`, so that LP recommendation preview is not confused with market-intent semantics.
20. As a reviewer, I want each slice to be independently demoable, so that academic reporting and supervisor feedback can happen incrementally.

## Implementation Decisions

- Delivery order is fixed: Slice 1 first, Slice 2 second.
- Slice 1 uses operator-facing read models instead of frontend-side aggregation.
- Slice 1 status semantics are canonical in backend and mapped to presentation labels in frontend.
- Slice 1 canonical status set is: `idle`, `prepared`, `running`, `completed`, `failed`.
- Slice 1 persists latest operator-visible state, not a full event log.
- Slice 1 persistence lives in a lightweight Postgres table dedicated to operator control status.
- Slice 1 record grain is `tenant_id + flow_type`.
- Allowed `flow_type` values are reserved upfront as: `weather_control`, `signal_preview`, `baseline_lp`, `gatekeeper`, `dispatch_execution`.
- Slice 2 `baseline_lp` read model shows forecast, feasible LP recommendation preview, projected SOC trace, and UAH economics.
- Slice 2 preview is expressed as hourly signed MW schedule, not `Proposed Bid` segments.
- Slice 2 economics use UAH as canonical currency.
- Slice 2 includes degradation as a UAH penalty component, not only as a KPI.
- Slice 2 battery simulation scope is a `Projected Battery State` simulator for LP preview.
- Slice 2 simulator state includes: `initial_soc`, `soc_min`, `soc_max`, `capacity_mwh`, `power_limit_mw`, `charge_efficiency`, `discharge_efficiency`, derived throughput, derived degradation penalty.
- Slice 2 simulator granularity is hourly to match `Level 1 Market Scope` for DAM.
- Slice 2 simulator starts from latest known persisted asset state / telemetry snapshot, with allowed scenario override for demo/offline mode.
- Slice 2 explicitly does not emit `Proposed Bid`, `Cleared Trade`, or `Dispatch Command` contracts.
- Multi-venue semantics, full digital twin physics, dispatch sandboxing, and universal dashboard abstraction are out of scope for these slices.

## Testing Decisions

- Good tests assert external behavior and stable contracts, not implementation details.
- Slice 1 should be covered by focused API tests for read models, persistence-backed status transitions, and dashboard integration behavior.
- Slice 2 should be covered by simulator tests, LP preview contract tests, and API/dashboard read-model tests.
- The preferred testing seam for backend is the existing FastAPI API slice and isolated Python modules for simulation logic.
- Frontend tests should focus on operator-visible outcomes: status labels, visible summaries, and chart/read-model rendering states.
- Prior art exists in the current FastAPI tests around [tests/api/test_main.py](d:/School/GoIT/Courses/Diploma/tests/api/test_main.py) and in the already validated dashboard control-plane flow.

## Out Of Scope

- Multi-venue bidding and venue-aware portfolio routing
- Explicit `Proposed Bid` generation in Slice 2
- `Cleared Trade` or settlement modeling in Slice 2
- Dispatch execution orchestration
- Full digital twin physics with temperature, SOH, SEI, or inverter behavior
- Full audit/event-sourcing subsystem for operator actions
- General-purpose dashboard platform abstractions before business slices are complete

## Further Notes

- This PRD intentionally preserves the glossary boundary from [CONTEXT.md](d:/School/GoIT/Courses/Diploma/CONTEXT.md): recommendation preview is not the same thing as market intent, cleared result, or physical execution.
- `gatekeeper` and `dispatch_execution` are reserved `flow_type` names, but their read models should not be implemented until later slices actually need them.
- Academic/demo readiness matters for Slice 1 completion and should be treated as a first-class acceptance target, not as optional polish.
