## Weekly report requirements

A weekly report should include:

- completed tasks and project changes
- identified risks and challenges
- next-week plan
- links to artifacts such as code, docs, screenshots, or demo materials

The report should stay compact and evidence-based.

## Literature review requirements

The literature review should not be a plain bibliography. It should:

- analyze strengths and weaknesses of existing approaches
- identify gaps in the current state of the art
- justify why the selected project methodology is needed
- connect sources to concrete architecture or methodology choices

Useful thematic grouping for this repo:

1. BESS arbitrage and market context
2. Electricity price forecasting
3. Predict-then-optimize vs decision-focused learning
4. Battery degradation and economic penalties
5. Orchestration and reproducible MLOps support

## Repo-specific writing rules

- Use Ukrainian by default.
- Do not claim that planned DFL components are already implemented.
- Distinguish clearly between:
  - verified MVP baseline
  - research-backed target architecture
- Keep domain language aligned with `CONTEXT.md`.
- For Week 1, treat the report and the literature review as separate but linkable artifacts.

## Common mistakes to avoid

- Writing a literature review as a source dump with no analysis
- Mixing implemented code with aspirational design without labeling the difference
- Repeating the same long prose in both weekly reports and thesis chapters
- Ignoring syllabus wording when deciding artifact scope or expected outputs
